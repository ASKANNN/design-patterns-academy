# Contributor Onboarding

Version: 1.0

---

# Purpose

This prompt is the standard entry point for anyone joining this project.

It ensures that every contributor starts from the same context, follows the same workflow, and respects the same architectural decisions.

This prompt should be used whenever:

* a new working session is started;
* a different tool or environment is used;
* the previous conversation or context has been lost;
* the project is resumed after a long break.

---

# Instructions

You are joining an existing software project.

Before performing any work, read the following repository documents in this exact order:

1. README.md
2. ROADMAP.md
3. docs/project-management/PROJECT_STATE.md
4. docs/project-management/ENGINEERING_GUIDE.md
5. docs/project-management/OWNER_GUIDE.md

Treat these files as the only source of truth.

Do not rely on previous conversations.

Do not make assumptions.

Do not reinterpret previous decisions.

---

# Your Responsibilities

After reading the documentation you must understand:

* the project vision;
* the repository architecture;
* the current development phase;
* the approved workflow;
* your role in the project;
* the coding standards;
* the documentation standards.

---

# Mandatory Rules

Always:

* respect the approved architecture;
* respect the existing folder structure;
* follow the current roadmap phase;
* follow the coding standards;
* follow the documentation standards;
* work efficiently and avoid unnecessary output;
* modify only the requested files;
* keep the repository consistent.

Never:

* redesign the architecture;
* rename folders;
* move files without approval;
* introduce alternative project structures;
* implement future roadmap phases;
* modify unrelated files;
* regenerate approved work.

---

# Working Procedure

After reading the documentation:

1. Confirm that you understand the project.
2. Wait for the next task.
3. Implement only the requested scope.
4. Stop after completing the task.
5. Wait for review or the next instruction.

Never continue automatically.

---

# Review Rules

If asked to review existing work:

* review only the requested scope;
* do not reject work because of future roadmap items;
* report only real problems;
* do not request unrelated changes.

---

# Implementation Rules

When implementing new functionality:

* reuse existing components whenever possible;
* keep the code modular;
* maintain consistency with the existing codebase;
* avoid unnecessary abstractions;
* avoid duplicate code;
* prioritize readability and maintainability.

---

# Communication Rules

Keep responses concise.

Do not repeat information already documented.

If additional improvements are discovered:

Report them separately.

Do not implement them automatically.

---

# Expected Response

After reading the documentation, respond only with:

> I have reviewed the project documentation and understand the current project state. I am ready for the next task.

Wait for further instructions.
