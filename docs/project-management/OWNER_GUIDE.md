# OWNER GUIDE

## Руководство владельца проекта

Версия: 1.0

---

# Назначение документа

Этот документ является главным руководством владельца проекта.

Его задача — помочь восстановить контекст проекта в любой момент времени независимо от:

* смены инструментов разработки;
* смены IDE;
* смены компьютера;
* длительного перерыва;
* смены разработчиков.

Если возникает вопрос **"что делать дальше?"**, сначала открывается именно этот документ.

---

# Главная философия проекта

Проект строится один раз.

Архитектура создаётся один раз.

Документация постоянно поддерживается в актуальном состоянии.

Контент развивается постепенно.

Мы не переписываем проект ради новых идей.

Мы постепенно расширяем уже существующую архитектуру.

Главное правило:

> Сначала качество. Потом количество.

# Работа с новым участником

Если к проекту подключается новый участник или новый инструмент разработки, нельзя сразу просить его писать код.

Сначала необходимо ввести его в проект.

Для этого используется onboarding.

Последовательность:

1. Новый участник читает документацию.
2. Участник подтверждает понимание проекта.
3. Только после этого выдаётся задача.

---

# Что дать новому участнику

Всегда использовать одинаковый порядок.

Участник должен прочитать:

* README.md
* ROADMAP.md
* PROJECT_STATE.md
* ENGINEERING_GUIDE.md

После этого участник должен ожидать следующую задачу.

Никогда не приступать к написанию кода до прохождения onboarding.

---

# Как проходит разработка

Каждая задача проходит одинаковый жизненный цикл.

1. Определяется следующая фаза.
2. Исполнитель реализует только эту фазу.
3. Проверяется сборка.
4. Проверяется работа приложения.
5. Выполняется ревью.
6. Если всё хорошо — APPROVED.
7. При необходимости обновляется документация.
8. Выполняется Commit.
9. Выполняется Push.
10. Начинается следующая фаза.

Никогда не перескакивать через этапы.

# Проверка перед ревью

Перед тем как показывать результат:

* проект собирается без ошибок;
* нет предупреждений;
* интерфейс работает;
* ничего не сломано;
* архитектура не нарушена.

Только после этого отправлять работу на ревью.

# Когда обновлять документацию

README

Только при серьёзных изменениях проекта.

ROADMAP

Только при изменении плана разработки.

PROJECT_STATE

После завершения каждой фазы.

ENGINEERING_GUIDE

Только если меняются общие инженерные правила работы над проектом.

OWNER_GUIDE

Только если меняется процесс разработки.

Остальная документация обновляется только тогда, когда это относится к её области ответственности.

---

# Когда делать Commit

Commit выполняется после успешного завершения работы.

Не коммитить код, который ещё не проверен.

Один логически завершённый этап = один понятный Commit.

---

# Когда делать Push

Push выполняется после:

* успешной сборки;
* ревью;
* завершённого Commit.

Не отправлять в репозиторий непроверенный код.

---

# Как не потеряться

Если прошло много времени:

Не открывать переписку.

Не пытаться вспомнить.

Открыть документацию.

Порядок восстановления:

1. README
2. ROADMAP
3. PROJECT_STATE
4. ENGINEERING_GUIDE
5. OWNER_GUIDE

После этого открыть дерево проекта.

Только потом приступать к работе.

---

# Если появилась новая идея

Не менять архитектуру сразу.

Сначала ответить себе:

Нужна ли эта идея прямо сейчас?

Если нет — записать её и вернуться после завершения текущего этапа.

---

# Что запрещено

Нельзя:

* менять архитектуру без причины;
* переименовывать папки;
* переносить файлы ради красоты;
* переписывать рабочий код без необходимости;
* ломать структуру проекта;
* хранить важные решения только в переписке.

# Экстренное восстановление проекта

Если проект открывается спустя месяцы или годы:

1. Не паниковать.
2. Не искать старые переписки.
3. Открыть README.
4. Открыть ROADMAP.
5. Открыть PROJECT_STATE.
6. Открыть ENGINEERING_GUIDE.
7. Открыть OWNER_GUIDE.
8. Ознакомиться с текущей фазой.
9. Пройти onboarding перед продолжением работы.
10. Продолжить работу с текущего состояния.

# Главное правило владельца проекта

Репозиторий — это память проекта.

Документация — это инструкция.

Инструменты разработки — это средство, а не источник решений.

Все решения должны быть зафиксированы в проекте.

Никогда не полагаться на память, историю переписки или случай.
